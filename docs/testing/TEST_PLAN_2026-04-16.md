## 1. Executive Summary

This is a comprehensive test plan covering all 12 feature areas of the eGhiseul.ro platform prior to Sprint 6 (Notifications & Polish). Sprint 4 and Sprint 5 delivered a large body of NEW / FIXED features that are still untested with real data: KYC S3 uploads, KYC face matching, KYC confidence tracking, Number Registry own page + document download, CLIENT_DETAILS_BLOCK legal format in contracts, multi-signature embedding, unified TVA 21%, and admin order detail polish. These receive **P0** priority.

### Totals

| Metric | Count |
|---|---|
| **Total test cases** | **228** |
| P0 (critical / must pass before release) | 96 |
| P1 (high — must pass before end of sprint) | 89 |
| P2 (regression / polish) | 43 |
| Feature areas covered | 12 |
| API endpoints covered | 60+ |
| Estimated manual effort (first pass) | 24-32 hours (3-4 QA-days) |
| Estimated automated coverage achievable | ~40% (Playwright + API specs) |

### Suggested execution order

1. **Smoke (≈1h)** — Auth login, service catalog loads, create-order wizard opens, admin dashboard loads, S3 upload returns 200.
2. **Critical paths (≈8h)** — P0 cases across Wizard PF + PJ flows, Payments, Document generation, Admin order workflow, KYC/Signature.
3. **Edge cases (≈8h)** — OCR street-prefix, diacritics, CNP checksum, CUI ANAF, locker geocoding, empty KYC confidence.
4. **Regression (≈6h)** — All P2 cases from Sprints 1-3.
5. **Security pass (≈4h)** — Rate limiting, RLS, IDOR, encryption verification.

## 7. Gaps Identified During Plan Creation

The 5 biggest testing gaps discovered:

1. **Contract legal validity (LEG-001 to LEG-011)** — never tested. Signature metadata (IP, UA, timestamp, SHA-256 hash) and consent tracking are live in production code but have no test coverage at all. Legal exposure if hash or consent does not actually persist.
2. **KYC confidence tracking (WIZ-018, WIZ-019, ADM-017 to ADM-022)** — brand-new feature introduced in Sprint 5. Per-document confidence scores flow through wizard state → DB → admin UI color coding. Zero tests. High risk since color thresholds (70/50) drive admin manual-review decisions.
3. **Multi-signature DOCX embedding (DOC-006 to DOC-009)** — DrawingML inline image injection via `signature-inserter.ts` is complex binary manipulation. No automated or manual tests verify client, company, lawyer signatures render correctly in the generated DOCX.
4. **CLIENT_DETAILS_BLOCK legal format parity (DOC-003, DOC-004, DOC-005, WIZ-029)** — preview uses same formatter as final DOCX, but nobody has compared them side by side for both PF and PJ variants. Lawyer has explicit format requirements for Barou.
5. **Number Registry document linking (REG-009, REG-010, REG-012, REG-021)** — registry-to-document FK relationship is critical for audit trail. Re-allocation-on-regeneration logic has never been verified, and FileDown icon + CSV "Document" column are brand-new UI pieces.

Secondary gaps: Stripe webhook idempotency (PAY-010), CRON_SECRET protection (COUR-014), S3 region enforcement, concurrent registry allocation race safety (REG-020).
